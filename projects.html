Venatio

Problem Statement

This application simulates NASA's Artemis II mission and helps users understand mission movement, data, and spacecraft behavior through an interactive technical model.

Technical Stack

  • Languages: HTML5, CSS3, JavaScript, GLSL
  • Tools: P5.js, React, GSAP, React Native, Expo

Process & Reflection

Our team used an Agile workflow to keep development flexible across the web simulation, mobile app, and informational website. We split the work into separate GitHub repositories, checked in regularly before committing major changes, and used feedback from student testers to improve usability, including adding menu tooltips after testers struggled without instructions. A major challenge was making Venatio run smoothly on low-powered and mobile devices. During debugging, we found memory issues caused by high-resolution textures, browser engine differences, and iOS WebView limitations. To solve this, we reduced texture quality from 8K to 4K on mobile, redesigned the menu for smaller screens, and ported the experience through React Native WebView, reducing RAM usage from about 1.5 GB to roughly 350 MB while keeping the simulation accessible.

Eduflash

Problem Statement

This platform supports accessible education by helping students from underrepresented communities discover and engage with learning resources for free.

Technical Stack

  • Languages: HTML5, CSS3, Javascript, Python, SQL
  • Tools: OpenAI API, PostgreSQL, AWS, Flask

Process & Reflection

We designed EduFlash around the idea that free education should be reliable, community supported, and accessible to anyone with basic internet access. Our process started with narrowing a large social problem into a platform where users could create guides, follow structured learning paths, rate resources, discuss accuracy, and use AI-assisted guide creation in multiple languages. We intentionally built the frontend with vanilla JavaScript so the platform would stay lightweight and efficient, then connected it to a Flask backend, PostgreSQL database, and AWS hosting. The biggest challenge was integrating several moving parts under a tight hackathon timeline. Debugging focused on making sure authentication, guide creation, database storage, and hosted deployment worked together securely. We overcame this by prioritizing core features, setting deadlines in Jira, and adapting quickly when scope or integration issues appeared.

Tavio

Problem Statement

This app helps visually impaired users order food more easily by reducing friction in menu navigation and selection.

Technical Stack

  • Languages: HTML5, CSS3, JavaScript, Python, SQL, Dart
  • Tools: Flutter, PostgreSQL, AWS

Process & Reflection

We designed Tavio around the problem that restaurant menus are overwhelmingly visual, which makes ordering slower and less independent for visually impaired users. Our design process focused on four core requirements: the app had to be accessible, fast to order from, simple to navigate, and reliable enough for a real restaurant setting. We built the mobile experience with Dart and Flutter, connected it to a PostgreSQL-backed system, and tested the ordering flow to find places where users could get confused or slowed down. One major challenge was turning a normally visual task into a clear step-by-step interaction without adding too much friction. Debugging was not just about fixing code errors; it also meant testing the user flow, identifying where the interface failed accessibility expectations, and refining the experience so ordering felt direct and usable.

TSA Engineering Design 2026

Problem Statement

This engineering design project proposes a microfluidic approach for detecting signs of life on Europa, one of Jupiter's moons.

Technical Stack

  • Languages: HTML5, CSS3, JavaScript, SCAD, C++
  • Tools: SCAD

Process & Reflection

Our design process started with researching the extreme constraints of Europa, including cryogenic ice, radiation, limited power, contamination risk, and the need for direct subsurface samples. We compared multiple concepts, including an ultrasonic-thermal probe, automated mapping drones, and a drill with an integrated microfluidic sampling chip. After weighing each option's advantages and drawbacks, we chose a combined MSES design because it provided direct ice penetration, real-time analysis, contamination control, and efficient sample movement in one system. A major challenge was turning a complex scientific idea into a feasible prototype plan. We worked through this by sketching system layouts, modeling the drill and chip in CAD, planning the circuit and fluid flow, and designing tests for sample movement, signal processing, durability, and collection accuracy. The project pushed us to think like engineers: define the problem, research existing technology, test assumptions, iterate from results, and refine the final design for compactness and long-term mission use.
